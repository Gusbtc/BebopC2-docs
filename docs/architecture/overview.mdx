# Architecture Overview

BebopC2 is an asynchronous C2 framework with a Go teamserver, browser operator console, and native C beacons for Windows and Linux. The current v1.5.0 architecture keeps the beacon core focused on transport, crypto, task dispatch, native built-ins, file transfer, session mode, SOCKS, and Windows BOF loading.

```text
Browser operator console
          |
          v
Go operator-client  ---> Go teamserver
                              |
                              v
                 Modular Windows and Linux beacons
```

## Component model

The teamserver owns state and task packing. The operator client is a thin UI over authenticated API calls and WebSocket events. Beacons execute tasks and return encrypted results.

### Key Components

| Component | Role |
| --- | --- |
| **Teamserver (Go)** | Auth, listeners, build pipeline, task queues, results, event log, library storage, and BOF/.NET packing. |
| **Windows beacon (C)** | WinHTTP transport, CNG crypto, native built-ins, file transfer, session mode, SOCKS, execute-assembly, and x64 COFF BOF loading. |
| **Linux beacon (C)** | musl/mbedTLS beacon with native Linux built-ins, file transfer, session mode, and SOCKS. |
| **Operator client (Go/JS)** | Browser console, sessions, listeners, build page, terminal, file browser, loot, library, chat, and event log. |

## Data Flow

- **Registration:** Beacons use RSA-2048 OAEP to establish a secure, authenticated session and exchange a unique AES-256 key.
- **Command & Control:** All subsequent traffic uses an **Encrypt-then-MAC** (AES-CBC + HMAC-SHA256) envelope with AES and HMAC subkeys derived by `HMAC-SHA256(master, label)`.
- **Tasking:** HTTP beacons drain up to 32 queued tasks per check-in. Session-mode beacons can receive tasks over a persistent TCP channel.
- **Modular Delivery:** `bof-execute` queues `TaskBOF` (`16`). `inline-assembly` also queues a BOF task containing the native loader object, managed bridge, target assembly, and arguments.

## Design rules

- Windows API calls in the beacon are resolved dynamically through generated hashes.
- Beacon strings are generated by `teamserver/obfgen` and decrypted at use sites.
- Teamserver libraries are separated: built-in BOFs in `teamserver/resources/bofs/`, operator BOFs in `~/.bebop/bofs/`, assemblies in `~/.bebop/assemblies/`.
- Command documentation must match actual output from code. If a command succeeds silently, docs should say so.
